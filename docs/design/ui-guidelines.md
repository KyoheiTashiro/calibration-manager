# UIガイドライン(業務用・情報density優先)

親: [domain-model.md](../domain-model.md) / [screen-design/README.md](../screen-design/README.md)

対象読者: フロントエンド実装者。本書はビジュアルデザインの規約集であり、画面ごとの表示項目・遷移・状態遷移などの仕様は [screen-design/](../screen-design/) が正であり本書では再定義しない。共通仕様(§0)と矛盾する記述がある場合は screen-design 側を優先する。

## 1. 設計方針

calibration-manager は**業務用の機器管理アプリ**である。

- **PC中心・情報密度優先**: 主利用シーンはオフィスでのPC操作。1画面で多くの行・列を俯瞰できることを大きい文字より優先する。モバイルは閲覧・簡易操作(通知確認、記録登録など)をサポートする第二の導線とする。
- **テーブルUI多用**: 機器・項目・案件の一覧が中核。ソート・フィルタ・行アクションを一貫したパターンで提供する。
- **ステータスの即座な視認性**: 期限切れ・要発注などのステータスは業務上のリスクに直結するため、色 + ラベルで一目で判別できることを最優先する(§4)。
- **控えめな演出**: アニメーション・装飾は最小限。業務ツールとして即応性・信頼性を優先し、派手な遷移効果は避ける。
- 標準的なWebの密度(ルート16px、Tailwind標準スケール)を基準とする。

## 2. タイポグラフィ

ルートフォントサイズは標準の `16px`(Tailwind既定スケールをそのまま使用。独自 `rem` 再定義はしない)。

| 要素                    | Tailwindスケール                   | 用途                                               |
| ----------------------- | ---------------------------------- | -------------------------------------------------- |
| 極小                    | `text-xs` (12px)                   | テーブル内の補助情報、タイムスタンプ               |
| 小                      | `text-sm` (14px)                   | テーブル本文、フォームラベル、既定の本文サイズ     |
| 標準                    | `text-base` (16px)                 | ダッシュボードのカード数値以外の本文、ボタンラベル |
| 見出しh2                | `text-lg` (18px)                   | セクション見出し、モーダルタイトル                 |
| 見出しh1                | `text-xl` 〜 `text-2xl` (20〜24px) | ページタイトル                                     |
| KPI数値(ダッシュボード) | `text-3xl` (30px)                  | 期限切れ件数等のサマリーカード                     |

- 書体: `"Noto Sans JP", system-ui, -apple-system, "Segoe UI", sans-serif`。明朝体は使用しない。
- 数値列(管理番号、件数、金額、日付)は `font-variant-numeric: tabular-nums` で桁を揃える。テーブルの可読性に直結するため必須。
- テーブル本文は既定 `text-sm` とする(1画面あたりの表示行数を優先)。

| 用途                           | 色                    | 備考                              |
| ------------------------------ | --------------------- | --------------------------------- |
| 本文テキスト                   | `#0F172A` (slate-900) |                                   |

ステータス表現色(赤・橙・青・黄・緑)は業務上の意味を持つ予約色として §4 のバッジ専用とし、UIの装飾目的では流用しない(誤読防止)。

## 4. ステータスバッジ(5色)

[screen-design/README.md §0.3](../screen-design/README.md#03-ステータスバッジ色共通定義) の5値・優先度・判定条件を UI トークンへ落とし込む。判定ロジック自体は本書では再定義しない。

| ステータス   | ラベル   | Tailwind (bg / text)            | 補助枠線                   |
| ------------ | -------- | ------------------------------- | -------------------------- |
| `overdue`    | 期限切れ | `bg-red-100 text-red-800`       | `border border-red-300`    |
| `orderNow`   | 要発注   | `bg-orange-100 text-orange-800` | `border border-orange-300` |
| `inProgress` | 校正中   | `bg-blue-100 text-blue-800`     | `border border-blue-300`   |
| `dueSoon`    | 期限接近 | `bg-yellow-100 text-yellow-800` | `border border-yellow-300` |
| `ok`         | 正常     | `bg-green-100 text-green-800`   | `border border-green-300`  |

- **色のみに依存しない**: バッジは常に「色 + 日本語ラベル文字列」をセットで表示する(例: 赤背景の中に「期限切れ」の文字)。色分けドットや絵文字のみでの表現(screen-design のワイヤーフレーム中の🔴🟠🟡🟢🔵表記はあくまで説明用の簡易記法)は実装では採用しない。
- 実装は `statusBadgeClass(status)` / `statusBadgeLabel(status)` の単一ヘルパに集約し(screen-design README §0.3 準拠)、コンポーネント側で色・文言を個別定義しない。
- `-800` 文字色 × `-100` 背景の組み合わせは概ね WCAG AA(4.5:1)を満たす設計値(例: `red-800`/`red-100` は目安約6.8:1)。実装時はコントラストチェッカーで各色を検証すること。特に `dueSoon`(黄)は素の黄色文字だと対白コントラストが低いため、必ず `yellow-800` の濃色文字 + `yellow-100` 背景の組み合わせを用い、単色の黄アイコンのみでの表現はしない。
- 通知ベルの未読件数バッジ(§0.1)はステータスバッジと異なる用途のため、`bg-red-600 text-white` の別トークンを用い、5色パレットとは混同しない。

## 5. 共通レイアウト

構成は [screen-design README §0.1](../screen-design/README.md#01-全体構成) に準拠。本書では見た目の仕様のみ定める。

- **サイドバー**: PC幅では左固定、幅 `w-60`(240px)。ロゴ/アプリ名 + ルート一覧(§0.2の9項目)。現在ルートは `bg-blue-50 text-blue-700` でハイライト。
- **ヘッダー**: 高さ `h-14`(56px)。左にハンバーガー(モバイルのみ表示、`md:hidden`)、右端に通知ベル + 未読バッジ(未読0で非表示、§0.1準拠)。
- **モバイル**: `md`(768px)未満でサイドバーを非表示にし、ハンバーガータップでオーバーレイ表示(`fixed inset-0` + 半透明背景 + `Esc`/オーバーレイクリックで閉じる。§0.5のモーダル挙動と同様の閉じ方に統一)。
- コンテンツ領域は `max-w-screen-2xl` を上限とし、テーブルが多いページでは横幅をできるだけ活用する(中央寄せ・余白重視レイアウトにはしない)。

## 6. テーブル・一覧

一覧画面の表示項目・フィルタ項目・行アクションは各画面設計([02](../screen-design/02-equipment-list.md) [05](../screen-design/05-inspection-item-list.md) [08](../screen-design/08-orders.md) 等)に従う。本書は共通の見た目とインタラクションパターンを定める。

- **行の高さ**: 密度優先で `h-10`〜`h-11`(40〜44px)。大型タップ領域は不要(マウス操作前提)。
- **配色**: ヘッダー行 `bg-slate-50 text-slate-600 text-xs font-medium`。偶数行の軽微なzebra(`bg-slate-50`)は任意、罫線 `border-slate-200` の横線のみで縦罫線は原則引かない。行ホバーは `hover:bg-slate-50`。
- **ソート**: ソート可能な列見出しはクリック可能にし、`aria-sort="ascending" | "descending" | "none"` を付与。現在のソート列に矢印アイコンを表示。既定ソートは各画面設計の指定に従う(例: 点検校正項目一覧は `nextDueDate` 昇順)。
- **フィルタ**: 表上部に横並びのセレクト群 + 「クリア」ボタン(§0.7のワイヤーフレーム踏襲)。自由入力の検索ボックスがある画面([機器一覧](../screen-design/02-equipment-list.md))はセレクト群の左に配置する。フィルタは即時反映(適用ボタンなし)。
- **数値・日付列**: 右揃え + `tabular-nums`。文字列列(名称等)は左揃え。
- **横スクロール**: 列数が多い一覧はテーブルを `overflow-x-auto` のコンテナで包み、ページ全体は横スクロールさせない(WCAG 1.4.10 のデータテーブル例外を踏まえた対応。§11参照)。管理番号など主キー列の横固定(sticky first column)は将来検討とし本フェーズでは必須にしない。
- **行アクション**: 行右端にテキストボタン群(「記録」「案件」「編集」等)。ボタン高さ `h-8`(32px)程度、`gap-2` で配置。
- **空状態**: [screen-design README §0.7](../screen-design/README.md#07-空状態共通方針) の方針(アイコン + 説明文 + 主要CTA)を全一覧で統一実装する。アイコンは `text-slate-300` の中型(32px程度)、説明文は `text-sm text-slate-500`。

## 7. フォーム入力・エラー表示

フォームは react-hook-form + `zodResolver` を全画面で統一使用する。

- ラベルは入力欄の上、必須項目は「◯◯ <span class="text-red-600">\*</span>」で明示。
- エラーは該当フィールド直下に `text-xs text-red-600` で表示し、入力枠は `border-red-500 aria-invalid:border-red-500` に切り替える。フィールドとエラー文言は `aria-describedby` で関連付ける。
- 送信ボタンはエラー有無で無効化しない([screen-design README §0.5](../screen-design/README.md#05-モーダルの共通挙動)準拠)。送信試行(`handleSubmit` の invalid コールバック)でエラーを表示し、最初のエラーフィールドへフォーカスを移す。
- ヘルプテキスト(未入力時の補足)は `text-xs text-slate-500` でエラーと同じ位置に表示し、エラー発生時はヘルプをエラーに差し替える(同時表示しない)。
- CSVインポート等のファイル系エラーは行番号付きの一覧表示とし、個別フィールドエラーと視覚的に区別する(赤背景の警告ボックス)。

## 8. モーダル・確認ダイアログ

挙動・発火条件は再定義せず、[screen-design README §0.5](../screen-design/README.md#05-モーダルの共通挙動)(モーダル共通挙動)・[§0.6](../screen-design/README.md#06-確認ダイアログポリシー)(確認ダイアログポリシー)を参照する。見た目の規約のみ以下に定める。

- 確認ダイアログ: `max-w-sm`、タイトル `text-base font-semibold`、破壊的操作の実行ボタンは §3 の危険色(`bg-red-600`)、キャンセルは中立(`border border-slate-300 text-slate-700`)。既定フォーカスは安全側(キャンセル)。
- フォーム系モーダル(項目編集・実施記録登録・案件作成・マスタ編集など): `max-w-lg`〜`max-w-2xl`。フッターは右寄せで「キャンセル」→「保存」の順。
- データ全削除モーダル([設定](../screen-design/11-settings.md))はチェックボックス同意までは実行ボタンを `disabled` の淡色表示にしてよい(§0.6の2段階確認を視覚的に表現する例外)。

## 9. 日付表示

[screen-design README §0.4](../screen-design/README.md#04-日付表示形式)準拠で `YYYY-MM-DD` に統一する。

- 表示・入力・CSV すべて同一フォーマット。相対表現(「3日前」等)は使用しない。
- 未実施・未設定は `—`(em dash 1文字)で表示し、空白のまま放置しない。
- テーブル内の日付列は等幅数字(§2)で揃え、`YYYY-MM-DD` の10桁固定幅で列がガタつかないようにする。

## 10. 操作フィードバック・余白

- ボタン高さ: 主要ボタン `h-9`(36px)、二次・行内ボタン `h-8`(32px)。大型ボタンは不要(マウス/トラックパッド操作前提)。モバイル表示時のみタップ領域を `min-h-11`(44px)に拡大する。
- 状態変化(保存・遷移)は150ms程度の短いトランジションのみ。派手なアニメーションは避ける。
- 保存成功はトースト通知(右下、3秒程度で自動消滅)または一覧の即時再描画で表現し、成功のたびにモーダルを出さない。
- ローディングはスケルトン行(テーブル)またはボタン内スピナーで表現し、画面全体のブロッキングオーバーレイは多用しない。

## 11. アクセシビリティ

WCAG 2.1 AA準拠を目標とする。

- コントラスト比: 通常テキスト4.5:1以上、UIコンポーネント・グラフィック3:1以上(§3・§4のトークンはこれを満たす設計値。実装時にツールで検証)。
- セマンティックHTML: `<table><thead><tbody><th scope="col">`、`<button>`、`<dialog>` またはARIA相当のロールを使用。
- フォーカスリング: `outline: 2px solid #2563EB; outline-offset: 2px`(`:focus-visible`)。
- キーボード操作: Tab順序は視覚順と一致。テーブル行アクションはボタンとして個別にフォーカス可能にし、行全体クリックのみに依存しない。
- 通知ベルの未読件数変化は `aria-live="polite"` の領域で告知する。
- `prefers-reduced-motion: reduce` 時はトランジション・アニメーションを実質無効化する。
- ズーム400%(WCAG 1.4.10 Reflow)まで横スクロール無しでレイアウト維持。ただし列数の多いデータテーブルは同SCの例外(2次元データの提示)に該当するため、テーブル本体はコンテナ内横スクロール(§6)で許容する。

## 12. Tailwind設定(参考)

```js
// tailwind.config.js
theme: {
  extend: {
    colors: {
      primary: '#2563EB',
      primaryHover: '#1D4ED8',
      danger: '#DC2626',
      line: '#E2E8F0',
      subBg: '#F8FAFC',
      // ステータスバッジは bg-{color}-100/text-{color}-800 のTailwind標準クラスをそのまま使用し、
      // 独自トークンは追加しない(色名との対応を保つため)
    },
    height: { rowSm: '2.5rem', rowMd: '2.75rem' }, // テーブル行高さ
  },
},
```

/* ベースCSS */
